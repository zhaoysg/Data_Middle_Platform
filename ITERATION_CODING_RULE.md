# 迭代编码规则

> 适用于本仓库后续所有二次开发与迭代编码。默认目标是 `ruoyi-*` 主工程，除非需求明确指向 `bgdata/*`，否则不要混改两套代码线。

## 1. 总原则

1. 先定位模块，再写代码；先确认联动面，再做修改。
2. 普通业务迭代坚持最小改动原则，不顺手重构公共底层，不顺手改另一条代码线。
3. 后端、前端、菜单权限、数据库结构、接口协议必须按一条业务链一起看，不能只改其中一层。
4. 新增能力优先复用项目现有模式，保持和当前工程一致，不额外引入新的分层习惯和技术栈。
5. 任何涉及权限、租户、数据权限、接口加解密、日志审计的改动，都按高风险改动处理，必须显式核对。

## 2. 生效范围

### 2.1 默认主线

- 后端主线：`ruoyi-admin`、`ruoyi-common/*`、`ruoyi-modules/*`、`ruoyi-extend/*`
- 前端主线：`ruoyi-vben-ui/apps/web-antd`

### 2.2 非默认范围

- `bgdata/*` 是仓库内并存的另一套工程
- 后续没有明确说明时，不要把 `bgdata/*` 当成 `ruoyi-*` 业务的一部分一起改

## 3. 开工前必须先回答的 8 个问题

1. 这次需求属于哪个模块：`ruoyi-admin`、`ruoyi-common`、`ruoyi-system`、`ruoyi-workflow`、`ruoyi-job`、`ruoyi-datasource`、`ruoyi-generator`、`ruoyi-extend`，还是前端 `web-antd`？
2. 这是普通 CRUD、流程业务、调度任务、认证改造，还是基础能力扩展？
3. 是否涉及数据库表或字段变更？
4. 是否涉及菜单、路由、权限码、按钮权限或角色授权？
5. 是否涉及租户隔离、数据权限、超管特判？
6. 是否涉及登录、密码、敏感字段、文件上传、导入导出、接口加解密？
7. 是否需要前后端同时改动才能形成完整闭环？
8. 改完后要如何验证：编译、类型检查、接口联调、页面点选、权限回归？

只要这 8 个问题里有 2 个以上答案是“是”，就不能只盯着单文件修改。

## 4. 模块落点规则

### 4.1 后端

- `ruoyi-admin`
  - 只放启动入口、认证入口、全局接入类逻辑
  - 登录、注册、统一认证链路、全局启动配置优先在这里处理
- `ruoyi-common/*`
  - 只放跨模块复用的公共能力
  - 如果一个能力只服务单个业务模块，不要为了“看起来通用”提前下沉到 common
- `ruoyi-modules/ruoyi-system`
  - 默认业务主战场
  - 常规后台管理页面、权限、字典、参数、租户、组织、用户、角色优先放这里
- `ruoyi-modules/ruoyi-workflow`
  - 审批流、流程定义、任务中心、流程业务挂接放这里
- `ruoyi-modules/ruoyi-job`
  - 调度执行器、补偿任务、汇总任务、SnailJob 执行逻辑放这里
  - 不要把普通后台 CRUD 页面塞进这个模块
- `ruoyi-modules/ruoyi-generator`
  - 只处理生成器相关功能，或者用它生成标准 CRUD 初稿
- `ruoyi-modules/ruoyi-datasource`
  - 数据源管理相关功能放这里
- `ruoyi-extend/*`
  - 监控后台、SnailJob 服务本体这类扩展服务放这里

### 4.2 前端

- 默认只在 `ruoyi-vben-ui/apps/web-antd` 迭代
- 没有明确理由，不要把主业务落到 `apps/web-antdv-next`
- 页面放 `src/views/**`
- 接口封装放 `src/api/**`
- 路由适配看 `src/router/access.ts`、`src/router/routes/core.ts`、`src/router/routes/local.ts`

## 5. 后端编码规则

### 5.1 分层必须对齐现有工程

- `Controller` 只做入参接收、权限注解、结果返回、少量编排
- `Service` 承担业务编排、事务、校验和跨表处理
- `Mapper` 承担数据访问，优先沿用 MyBatis-Plus + 现有 Mapper 模式
- `Entity / Bo / Vo` 分开维护，不要混用一个类兼顾请求、持久化、返回

### 5.2 统一返回与分页方式

- 普通接口返回 `R<T>`
- 分页接口返回 `TableDataInfo<T>`
- 控制层优先复用 `BaseController#toAjax`

### 5.3 沿用当前常见注解组合

- 查询/写入权限：`@SaCheckPermission`
- 写操作幂等防重：`@RepeatSubmit`
- 操作审计：`@Log`
- 参数校验：`@Validated`
- 敏感接口或登录密码类接口：按现有模式评估 `@ApiEncrypt`
- 事务写操作：按现有模式使用 `@Transactional(rollbackFor = Exception.class)`

### 5.4 数据与权限硬约束

- 任何用户、角色、部门、岗位、租户相关写操作，都要核对数据权限和超管限制
- 自定义查询如果绕开了现有 Mapper 默认方法，要确认是否仍然带上了 `@DataPermission`
- 不允许为了赶进度跳过唯一性校验、数据范围校验、状态校验
- 涉及租户的接口，必须确认 `TenantHelper` 相关行为不会被破坏

### 5.5 对象与工具使用约束

- BO 转 Entity、Entity 转 VO 优先沿用现有 `MapstructUtils` / `@AutoMapper`
- 查询条件优先沿用 `Wrappers`、`LambdaQueryWrapper`、`LambdaUpdateWrapper`
- 不手写一套新的返回体、分页体、异常体、工具类风格

## 6. 前端编码规则

### 6.1 请求层规则

- 所有业务请求统一走 `src/api/request.ts` 的 `requestClient`
- 不直接在页面里裸写 axios 请求
- 如果后端接口启用了加密，前端调用必须显式传 `{ encrypt: true }`
- 下载、导入、原始响应等特殊场景，沿用现有 `isTransformResponse`、`responseType`、`ContentTypeEnum` 模式

### 6.2 页面层规则

- 优先复用现有 Vben 组件模式：`Page`、`useVbenVxeGrid`、`useVbenDrawer`、`useVbenModal`
- 列表页、详情弹层、导入弹层、重置密码弹层等结构，优先参考当前 `system/user` 现成写法
- 权限按钮必须挂 `v-access:code`

### 6.3 路由与菜单硬规则

- 本项目不是纯前端静态路由项目，后台业务页面默认由 `sys_menu` + `/system/menu/getRouters` 驱动
- 新增菜单型页面时，必须同时保证下面四件事成立：
  1. 后端有对应接口
  2. 前端有对应 `src/api/**`
  3. 前端有对应 `src/views/**`
  4. `sys_menu.component` 能正确映射到 Vue 文件路径
- 隐藏页、详情页、高亮回填页，如有必要，同步维护 `src/router/access.ts` 里的 `routeMetaMapping`

## 7. 典型联动规则

### 7.1 新增一个后台业务页面

至少同时检查以下项目：

1. 后端模块落点是否正确
2. 数据表、字段、索引是否齐全
3. `Bo / Vo / Entity / Mapper / Service / Controller` 是否成套
4. 前端 `src/api` 是否已补齐
5. 前端 `src/views` 是否已补齐
6. `sys_menu`、权限码、角色授权是否已补齐

### 7.2 修改一个已有页面

- 先确认这是页面展示问题、接口结构问题，还是权限问题
- 如果后端字段名、字典值、状态值、枚举值改了，前端展示和筛选条件必须同步改
- 如果只改页面，不要顺手改公共请求层或基础布局

### 7.3 修改认证、密码、用户、角色、租户

- 这类需求默认是高风险需求
- 必须同时检查：权限码、数据权限、超管保护、租户隔离、加解密、登录态影响

### 7.4 修改流程业务

- 工作流需求优先放 `ruoyi-workflow`
- 业务单据、流程定义、流程实例、任务中心的联动关系必须一起核对
- 不要把流程业务拆散到 system 和 workflow 两边随意混放

### 7.5 修改任务调度

- 任务执行逻辑优先写在 `ruoyi-job`
- 页面入口和运维展示不等于任务本体逻辑
- 不能把调度执行器当普通 CRUD 管理页去设计

## 8. 禁止事项

- 禁止默认混改 `ruoyi-*` 和 `bgdata/*`
- 禁止在普通业务需求里顺手重构 `ruoyi-common`
- 禁止只改前端页面、不补权限菜单就宣称功能完成
- 禁止只改数据库字段、不补 BO/VO/查询条件/表单校验
- 禁止绕开 `requestClient` 另起一套请求封装
- 禁止把密码、密钥、生产连接信息明文写进仓库代码或文档
- 禁止为了“快速可用”去掉权限校验、租户隔离、数据权限或接口加密

## 9. 交付前检查清单

1. 受影响模块是否定位正确，没有越界改动？
2. 前后端是否都能形成闭环？
3. 菜单、权限码、角色授权是否补齐？
4. 租户、数据权限、超管逻辑是否被验证？
5. 写操作是否具备日志、幂等、事务等必要保护？
6. 后端是否至少完成受影响模块编译验证？
7. 前端是否至少完成受影响应用类型检查或关键页面联调？
8. 如果改了模块边界、表结构、路由、菜单，是否同步更新项目文档？

## 10. 文档同步规则

- 修改项目结构、模块职责、启动方式，更新 `PROJECT_RULES.md`
- 修改模块地图、核心表、前端路由、二开入口，更新 `SECONDARY_DEV_MAP.md`
- 修改 `bigdata` 数据源、`sys_datasource`、外部数据源接入规则，更新 `BIGDATA_DATASOURCE_RULES.md`
- 新增长期有效的编码约束，优先更新本文件

## 11. 默认执行口径

后续任何迭代任务，如无额外说明，一律按下面的口径执行：

1. 先基于 `ruoyi-*` 主线分析需求落点
2. 按最小闭环补齐后端、前端、菜单权限、数据结构
3. 保持现有分层、命名、返回体、请求封装和动态路由机制不变
4. 完成必要验证后再交付

如果未来需求与本规则冲突，以“明确业务需求”优先；但在落代码前必须先指出冲突点，再决定是否突破本规则。
